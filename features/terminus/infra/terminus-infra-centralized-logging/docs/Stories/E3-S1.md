# E3-S1 — ArgoCD ServiceMonitor Fix: Metrics and Namespace Label

## Story

**As an** operator,  
**I want** ArgoCD to expose Prometheus metrics via ServiceMonitors,  
**so that** the 3 existing ArgoCD alert rules and the ArgoCD delivery dashboard have data.

## Context

The `monitoring-servicemonitors.yaml` comment says "ServiceMonitors for ArgoCD are managed via their respective Helm values" — but this was never implemented. ArgoCD Helm values have no `metrics` or `serviceMonitor` stanzas, so the 3 existing alert rules (`ArgoCDAppDegraded`, `ArgoCDAppOutOfSync`, `ArgoCDSyncFailureRateHigh`) and the ArgoCD delivery dashboard have been no-ops since prometheus was deployed.

This story is independent of Loki — it can be done in Sprint 1 in parallel with E1 work.

**Relevant architecture sections:**
- `architecture.md` §Component Decisions §5 — ArgoCD ServiceMonitor Fix

## Implementation Tasks

### Task 1: Update ArgoCD Helm Values

Update `platforms/k3s/helm/argocd/values.yaml` — add metrics and serviceMonitor stanzas:

```yaml
controller:
  metrics:
    enabled: true
    serviceMonitor:
      enabled: true
      additionalLabels:
        release: prometheus

server:
  metrics:
    enabled: true
    serviceMonitor:
      enabled: true
      additionalLabels:
        release: prometheus

repoServer:
  metrics:
    enabled: true
    serviceMonitor:
      enabled: true
      additionalLabels:
        release: prometheus

applicationSet:
  metrics:
    enabled: true
    serviceMonitor:
      enabled: true
      additionalLabels:
        release: prometheus
```

**Verify the label selector:** Check what label Prometheus uses to find ServiceMonitors in your cluster:
```bash
kubectl get prometheus -n monitoring -o jsonpath='{.items[0].spec.serviceMonitorSelector}' | python3 -m json.tool
```

If the selector is `matchLabels: {release: prometheus}`, the `additionalLabels` above are correct. Adjust if different.

### Task 2: Add Namespace Label

The `argocd` namespace needs the `observability.terminus.io/scrape: "true"` label so that namespace-based ServiceMonitor selectors include it.

Find the argocd namespace manifest (likely in `platforms/k3s/k8s/` or referenced by `argocd-infra.yaml`):

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: argocd
  labels:
    observability.terminus.io/scrape: "true"
```

If the namespace is not managed via a manifest, apply the label imperatively and note it for future infra-as-code reconciliation:
```bash
kubectl label namespace argocd observability.terminus.io/scrape=true
```

### Task 3: Sync and Verify

```bash
# Sync argocd ArgoCD app
argocd app sync argocd

# Wait for rollout
kubectl rollout status deployment -n argocd --timeout=3m

# Verify ServiceMonitors created
kubectl get servicemonitors -n argocd

# Verify metrics scrape target appears in Prometheus
# Go to Prometheus UI > Status > Targets and search for argocd
# Or:
kubectl port-forward -n monitoring svc/prometheus-operated 9090:9090 &
curl -s http://localhost:9090/api/v1/targets | python3 -m json.tool | grep argocd
```

### Task 4: Verify Alert Rules Have Data

```bash
# Check argocd_app_info is present
curl -s http://localhost:9090/api/v1/query?query=argocd_app_info | python3 -m json.tool | head -30

# Check existing alert rules are evaluatable
curl -s http://localhost:9090/api/v1/rules | python3 -m json.tool | grep -A5 "ArgoCDApp"
```
