# E1-S1 — Synology S3 Credentials: Vault + ExternalSecret Setup

## Story

**As an** operator,  
**I want** S3 credentials for Synology NAS stored in Vault and materialized as a Kubernetes Secret,  
**so that** Loki can authenticate to the S3 backend without credentials in git.

## Context

Loki prod uses Synology NAS S3-compatible API as the object storage backend (`nas.trantor.internal`, bucket `terminus-loki-prod`). Credentials must be stored in Vault KV and materialized via ESO as a Kubernetes Secret in the `monitoring` namespace. This is a Phase 0 prerequisite — nothing in E1 can deploy without it.

**Relevant architecture sections:**
- `architecture.md` §Phase 0 — Prerequisites
- `architecture.md` §Component Decisions §1 (Storage section)

## Pre-Implementation Checklist

- [ ] Verify Synology NAS hostname resolves: `nslookup nas.trantor.internal` from a cluster pod
- [ ] Confirm Synology S3-compatible API is enabled on the NAS (DSM > S3 Server package or compatible)
- [ ] Confirm bucket `terminus-loki-prod` does not already exist (or create if absent)
- [ ] Confirm Vault is accessible and KV secrets engine at `secret/` is enabled

## Implementation Tasks

### Task 1: Synology NAS Setup (manual/out-of-band)
1. Open Synology DSM > S3 Server (or compatible S3 API extension)
2. Create bucket: `terminus-loki-prod`
3. Create IAM-style service account/access key for Loki
4. Record `access_key_id` and `secret_access_key`

### Task 2: Store Credentials in Vault
```bash
vault kv put secret/terminus/loki/s3-creds \
  access_key_id=<LOKI_ACCESS_KEY> \
  secret_access_key=<LOKI_SECRET_KEY>
```

Verify:
```bash
vault kv get secret/terminus/loki/s3-creds
```

### Task 3: Create ExternalSecret Manifest

Create `platforms/k3s/k8s/monitoring/loki-s3-creds-es.yaml`:

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: loki-s3-creds
  namespace: monitoring
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: loki-s3-creds
    creationPolicy: Owner
  data:
    - secretKey: access_key_id
      remoteRef:
        key: secret/terminus/loki/s3-creds
        property: access_key_id
    - secretKey: secret_access_key
      remoteRef:
        key: secret/terminus/loki/s3-creds
        property: secret_access_key
```

**Note:** Adjust `ClusterSecretStore` name to match your ESO configuration (check `kubectl get clustersecretstore`).

### Task 4: Deploy and Verify
```bash
# Apply manifest (or let ArgoCD pick it up)
kubectl apply -f platforms/k3s/k8s/monitoring/loki-s3-creds-es.yaml

# Verify sync
kubectl get externalsecret loki-s3-creds -n monitoring
kubectl get secret loki-s3-creds -n monitoring
```

### Task 5: Verify S3 Connectivity
```bash
# From a pod in monitoring namespace — verify NAS S3 endpoint is reachable
kubectl run -n monitoring s3-test --image=curlimages/curl --restart=Never --rm -it -- \
  curl -v http://nas.trantor.internal/terminus-loki-prod/
```

Expected: 403 Forbidden or ListBucketResult (not connection refused).

## Acceptance Criteria

- [ ] Synology NAS S3 service account created; bucket `terminus-loki-prod` created
- [ ] Credentials stored in Vault KV at `secret/terminus/loki/s3-creds`
- [ ] ExternalSecret manifest at `platforms/k3s/k8s/monitoring/loki-s3-creds-es.yaml` committed
- [ ] `loki-s3-creds` Secret present in `monitoring` namespace (`kubectl get secret loki-s3-creds -n monitoring`)
- [ ] S3 endpoint reachable from within cluster (no connection refused)
