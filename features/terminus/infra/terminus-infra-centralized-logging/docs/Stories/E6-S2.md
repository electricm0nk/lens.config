# E6-S2 — VM Log Stream Validation and File Permission Hardening

## Story

**As an** operator,  
**I want** to validate VM log streams in Loki and harden Vault audit log file permissions,  
**so that** VM logs are confirmed flowing and the audit log access surface is minimized.

## Context

Validation + security hardening story. File permission hardening for Vault audit log is a security requirement from the architecture (`architecture.md` §Security Considerations §VM-Side Alloy).

**Pre-req:** E6-S1 complete (Alloy running on both VMs).

## Implementation Tasks

### Task 1: Verify Log Streams in Grafana

Run in Grafana Explore:
```logql
{service="vault", log_type="audit"} | tail 20
{service="postgres", log_type="db"} | tail 20
```

### Task 2: Verify Position File Persistence

```bash
# On vault VM
sudo systemctl restart alloy
sleep 30
# Check Loki for duplicate entries (compare timestamps)
# Alloy position file should record last read position
sudo cat /var/lib/alloy/positions.yaml
```

### Task 3: Vault Audit Log File Permission Hardening

Update `deploy-alloy-vm.yaml` to include:

```yaml
- name: Add alloy user to vault group
  user:
    name: alloy
    groups: vault
    append: yes

- name: Set vault audit log permissions
  file:
    path: /var/log/vault/audit.log
    owner: vault
    group: alloy
    mode: "0640"
```

Verify on vault VM:
```bash
ls -la /var/log/vault/audit.log
# Expected: -rw-r----- vault alloy
```

## Acceptance Criteria

- [ ] LogQL `{service="vault", log_type="audit"}` returns Vault audit events in Grafana
- [ ] LogQL `{service="postgres", log_type="db"}` returns Postgres log entries
- [ ] Alloy file position persists — no duplicate entries after `systemctl restart alloy`
- [ ] Vault audit log: `chmod 640`, `chown vault:alloy` — applied via Ansible playbook
- [ ] `alloy` user is member of `vault` group on `vault.trantor.internal`
